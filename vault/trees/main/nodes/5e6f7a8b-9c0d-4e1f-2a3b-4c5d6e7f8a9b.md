# Intent-/Natural-Language-Interfaces (LLM)

## Prinzip

Nutzer formuliert das Ziel, das System waehlt die Aktionsfolge. Skaliert gut ueber unbekannte Aktionen.

## Staerken

- Kein Vorwissen ueber Systemstruktur noetig
- Natuerliche Sprache als universelles Interface
- Gut fuer Gelegenheitsnutzer (SAP alle 3 Monate)
- Kann komplexe Workflows in einem Satz ausloesen

## Schwaechen

- **Nicht-deterministisch:** Gleiche Eingabe kann verschiedene Aktionen ausloesen
- **Langsamer pro Einzelaktion** als Tastenkuerzel/T-Code
- **Schwache Auditierbarkeit:** Die Aktions-Ableitung ist schwer reproduzierbar
- **Vertrauensproblem:** Bei Finanzbuchungen existenziell — Nutzer muss kontrollieren koennen

## SAP-Vergleich

SAP hat mit "Joule" einen AI-Assistenten eingefuehrt, aber als Overlay, nicht als Ersatz fuer T-Codes. Das ist bezeichnend: In Enterprise-Kontexten ist Determinismus zu wichtig, um ihn aufzugeben.

## Design-Anforderung fuer AI ERP

NL-Interface als **Einstiegshilfe und Long-Tail-Zugang**, nicht als primaere Interaktion:

- "Erstelle eine Bestellung fuer Lieferant X" → System zeigt Preview der Aktionsfolge
- **Plan-vs-Apply-Muster:** LLM schlaegt vor, Nutzer bestaetigt
- Deterministische Aktionen unter der Haube (LLM waehlt nur die richtige Aktion aus)
- Audit-Log zeigt sowohl den Intent als auch die ausgefuehrte Aktion
- Fallback: Nutzer kann jederzeit zur Command Palette wechseln

## Offene Frage

Wie garantiert man **Reproduzierbarkeit** bei LLM-basierten Aktionsableitungen? Ansaetze:
- Function Calling mit festem Aktionskatalog (LLM waehlt nur, fuehrt nicht aus)
- Logging des gewaehlten Aktionspfads als deterministische Replay-Einheit
- Temperatur 0 + Seed fuer maximale Konsistenz
