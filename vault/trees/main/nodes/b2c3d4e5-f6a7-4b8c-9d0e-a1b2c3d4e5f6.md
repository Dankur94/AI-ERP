# Grundprinzip & Hub-and-Spoke

OpenClaw folgt einer **Hub-and-Spoke-Architektur** mit einem zentralen Gateway als Control Plane. Die entscheidende Architekturidee: **Interface-Layer** (wo Nachrichten herkommen) und **Assistant Runtime** (wo Intelligenz und Ausfuehrung leben) sind strikt getrennt.

## Das Modell

```
Spokes (Eingaenge)          Hub (Zentral)           Runtime (Verarbeitung)
┌──────────┐
│ WhatsApp │──┐
├──────────┤  │  ┌──────────────┐   ┌────────────────┐
│ Slack    │──┼──│   Gateway    │───│  Agent Runtime  │
├──────────┤  │  │  (WebSocket) │   │  (LLM + Tools)  │
│ Web-UI   │──┘  └──────────────┘   └────────────────┘
├──────────┤         Port 18789
│ CLI      │──┘
└──────────┘
```

## Konsequenz
Man bekommt einen **persistenten Assistenten**, erreichbar ueber beliebige Messaging-Apps, mit zentral gemanagtem State. Kein Neustart der Session bei Kanalwechsel.
