# Stripe — Architektur-Lessons

| Stripe-Prinzip | Bedeutung fuer AI ERP |
|---|---|
| Monolith + Tooling statt Microservices | Produkte in einem Codebase halten, geteiltes Datenmodell ist wertvoller als unabhaengige Deploys |
| API-Versionierung als Burggraben | Kunde pinnt seine Version, Business-Logik nur fuer neueste Version, Transformation rueckwaerts — nie etwas brechen |
| Jedes neue Produkt folgt den Daten | Nicht brainstormen, sondern beobachten: welcher Schmerz taucht am Rand deines bestehenden Netzwerks auf? |
| Enterprise = anderes Packaging, nicht anderes Produkt | Selbe Plattform, andere Preise/Support/Config — kein separater Enterprise-Fork |
| Progressive Complexity via Packaging | Einfacher Modus versteckt Komplexitaet, Nutzer graduiert ohne Migration |

**Kontrast zu Odoo:** Odoo = viele Module lose gekoppelt. Stripe = ein Monolith eng gekoppelt. Beide erfolgreich. Entscheidend ist: **geteiltes Datenmodell**, egal ob Monolith oder Module.

**Referenz:** 7 Zeilen Code (2011) → Ruby-Monolith mit 15M Zeilen (2017) → 50% der Fortune 100 (2024)
