# Engineering-Eigenschaften

## Automatisierbarkeit / Skriptbarkeit

Kann ich die Bedienung programmatisch, als Batch oder im Scheduler fahren?

- **Unix:** Gewinnt hier haushoch — alles ist skriptbar
- **SAP-GUI:** Notorisch schlecht (weswegen BAPIs/RFCs nachgeschoben wurden)
- **IBM CICS:** Batch ueber JCL, aber fragil

Eine Paradigma-Entscheidung, die oft vergessen wird, bis man 10.000 Vorgaenge automatisieren will.

### Design-Implikation AI ERP
- API-first: Jede UI-Aktion hat ein API-Aequivalent
- CLI-Tool fuer Batch-Operationen
- Webhook-basierte Automatisierung

## Reversibilitaet / Undo

Wie leicht kann eine Aktion rueckgaengig gemacht werden?

- **Event-Sourcing-Systeme:** Strukturell im Vorteil (der Log ist die Wahrheit)
- **Transaktionale Systeme (SAP):** Oft nur Kompensationsbuchungen (Storno statt Undo)
- **Git:** Perfektes Undo-Modell (jeder Commit revertierbar)

Undo ist einer der unterschaetztesten UX-Faktoren ueberhaupt — wer Undo hat, traut sich zu experimentieren.

### Design-Implikation AI ERP
- Event Sourcing fuer geschaeftskritische Domaenen
- Explizites Undo fuer alle Benutzeraktionen
- "Papierkorb"-Semantik statt hartem Loeschen

## Auditierbarkeit / Nachvollziehbarkeit

Wer hat wann was gemacht, und warum? Im Enterprise-/Compliance-Kontext (SOX, GoBD) oft der entscheidende Faktor.

- **T-Codes:** Historisch stark, weil jede Transaktion geloggt wird
- **LLM-Intent-Interfaces:** Schwach, weil die Aktions-Ableitung schwer reproduzierbar ist
- **Event Sourcing:** Sehr stark (der Log IST der Zustand)

### Design-Implikation AI ERP
- Immutable Audit-Log als Kern-Infrastruktur
- Jede Aktion: Wer, Wann, Was, Warum (Intent + Ergebnis)
- LLM-Aktionen: Sowohl Prompt als auch gewaehlte Aktion loggen

## Observability

Laufende Systeme verstehen — nicht nur Logs, sondern auch "warum macht das System gerade das".

- **Imperativ:** Einfacher (Kausalkette explizit)
- **Deklarativ/Reaktiv:** Schwieriger (Kausalkette weniger explizit)
- **Event-driven:** Gut, wenn Event-Tracing implementiert (distributed tracing)

### Design-Implikation AI ERP
- OpenTelemetry-basiertes Tracing von Anfang an
- Event-Flow-Visualisierung im Admin-Dashboard
- Korrelation von Business-Events mit System-Metriken

## Determinismus / Reproduzierbarkeit

Gleiche Eingabe → gleiche Ausgabe?

- **T-Codes:** Deterministisch
- **LLM-Interfaces:** Nicht deterministisch
- **Deklarative Systeme:** Deterministisch (gleicher Zielzustand)
- **Event Sourcing:** Reproduzierbar (Events replayed → gleicher Zustand)

Bei Finanzbuchungen existenziell, bei kreativen Aufgaben egal.

### Design-Implikation AI ERP
- Geschaeftskritische Aktionen: Immer deterministisch
- LLM-Vorschlaege: Preview + Bestaetigung, nie blind ausfuehren
- Reproduzierbare Berichte: Gleiche Parameter → gleicher Report
