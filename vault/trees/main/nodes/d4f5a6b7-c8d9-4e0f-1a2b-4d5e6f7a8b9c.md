Taktisches Design beantwortet: "Wie strukturiere ich Code INNERHALB eines Moduls?"

**Entity** — Objekt mit eindeutiger Identitaet. Zwei Rechnungen mit gleichem Betrag sind trotzdem verschiedene Rechnungen (unterschiedliche ID).
Beispiel: `Rechnung { id: "RE-2026-0042", betrag: 1500, status: "offen" }`

**Value Object** — Objekt OHNE Identitaet. Nur der Wert zaehlt. Zwei Geldbetraege von 100 EUR sind austauschbar.
Beispiel: `Geld { betrag: 100, waehrung: "EUR" }` — kein braucht eine ID.

**Aggregate** — Eine Gruppe von Entities/Value Objects die zusammengehoeren und gemeinsam konsistent bleiben muessen. Hat einen "Aggregate Root" als Eintrittspunkt.
Beispiel: `Auftrag` (Root) → `Auftragsposition[]` → `Preis`. Man aendert nie eine Position direkt, sondern immer ueber den Auftrag.

**Repository** — Kapselt den Datenzugriff. Der Rest des Codes weiss nicht ob die Daten aus einer DB, Datei oder API kommen.
Beispiel: `AuftragRepository.findById("AU-123")` — ob das aus PostgreSQL oder einer JSON-Datei kommt, ist egal.

**Domain Service** — Logik die zu keiner einzelnen Entity gehoert.
Beispiel: `Preisberechnung.berechne(auftrag, kundenrabatt, steuerzone)` — das gehoert weder zum Auftrag noch zum Kunden.

**In eurer LeanHierarchy-App:**
- Entity: Ein Node (hat UUID, Titel, Kinder)
- Value Object: Ein WordMark (nur Index + nodeId, kein eigene Identitaet)
- Aggregate: Node + seine .md-Datei + Marks + Highlights (gehoeren zusammen)
- Repository: `readTree()`, `readNodeContent()`, `saveHighlights()` in main.js
