# Build-Begleitung — Mit Claude steuern

## Was

Du baust MIT dem Kunden und Claude. Du steuerst die Richtung, Claude schreibt den Code, der Kunde versteht was passiert.

## Warum Claude das nicht kann

- Claude sagt nie "Stopp, das ist der falsche Ansatz"
- Claude baut was man fragt — auch wenn es Unsinn ist
- Claude hat kein Urteilsvermoegen ueber Geschaeftslogik

## Deine Rolle im Build

| Claude macht | Du machst |
|---|---|
| Code schreiben | Entscheiden **welcher** Code |
| Bugs fixen | Entscheiden ob es ein Bug oder ein Feature-Missverstaendnis ist |
| Tests schreiben | Definieren **was** getestet werden muss |
| Schnell iterieren | **Wann stoppen** und nachdenken |
| Alles bauen was man fragt | Sagen: "Das brauchst du nicht" |

## Format

- **Woechentliche Build-Sessions** (2-3h) mit dem Kunden
- Du bringst die Architektur-Entscheidungen mit
- Ihr baut gemeinsam mit Claude
- Der Kunde lernt dabei sein eigenes System zu verstehen
- Du dokumentierst Entscheidungen fuer spaeter

## Der versteckte Wert

Der Kunde wird **muendiger**. Nach 3 Monaten kann er kleine Aenderungen selbst mit Claude machen. Grosse Aenderungen brauchen weiterhin dich. Das ist kein Bug — das ist das Geschaeftsmodell.

## Anti-Pattern vermeiden

- NICHT: "Ich baue alles, du wartest" → Abhaengigkeit, kein Verstaendnis
- NICHT: "Mach alles selbst mit Claude" → Insellosungen, kein System
- SONDERN: "Wir bauen zusammen, ich steuere, du lernst" → Nachhaltigkeit
