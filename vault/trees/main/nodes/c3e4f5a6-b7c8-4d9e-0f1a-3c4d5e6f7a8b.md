Strategisches Design beantwortet: "Wie schneide ich mein System in Teile?"

**Bounded Context** — Ein abgegrenzter Bereich, in dem Begriffe eine eindeutige Bedeutung haben.

Beispiel: "Kunde" bedeutet in der Buchhaltung etwas anderes als im CRM:
- Buchhaltung-Kunde = Rechnungsadresse, Steuernummer, Zahlungsziel
- CRM-Kunde = Kontaktperson, letzter Anruf, Lead-Score

Das sind zwei Bounded Contexts. Jeder hat sein eigenes Modell von "Kunde". Und das ist OK!

**Context Map** — Wie die Bounded Contexts miteinander kommunizieren:
- Shared Kernel: Zwei Kontexte teilen sich ein Stueck Code (gefaehrlich, aber manchmal noetig)
- Customer/Supplier: Ein Kontext liefert Daten, der andere konsumiert
- Anti-Corruption Layer: Ein Kontext schuetzt sich vor dem Modell eines anderen (z.B. SAP-Daten werden am Rand uebersetzt)

**Fuer AI ERP:**
Jedes Modul in `modules/<name>/` IST ein Bounded Context. Das ist bereits richtig angelegt laut CLAUDE.md. Entscheidend: Module kommunizieren ueber definierte Schnittstellen (Events, APIs), NICHT durch direkten Datenbankzugriff.
