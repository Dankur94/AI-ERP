# Memory-System

## Das Problem: Endliches Kontextfenster
Jedes LLM hat ein begrenztes Kontextfenster. Wenn eine Konversation laenger laeuft, muss irgendwann aelterer Kontext weichen. OpenClaws Loesung:

## Die Komprimierungsstrategie

```
Phase 1: Volle Konversationshistorie (aktuelle Session)
    ↓ Kontextfenster wird voll
Phase 2: Aeltere Nachrichten → kompakte Zusammenfassungen
    ↓
Phase 3: Aktuelle Nachrichten bleiben intakt
    ↓
Phase 4: Zusammenfassungen werden in JSONL persistiert
```

## Der Workspace: Git-backed Markdown
Der Arbeitsbereich ist ein **git-backed Verzeichnis** aus einfachen Markdown-Dateien:
- **Lesbar**: Jeder kann die Memory-Dateien oeffnen und lesen
- **Versionierbar**: Git-History zeigt, was sich wann geaendert hat
- **Portierbar**: Keine proprietaere Datenbank — einfach kopieren
- **Diffbar**: Aenderungen im Memory sind als Git-Diffs nachvollziehbar

## Kein Lock-in
Im Gegensatz zu proprietaeren Systemen (SAPs Datenbank-Abhaengigkeit) kann man:
- Memory zwischen Instanzen migrieren
- Memory manuell editieren
- Memory in andere Systeme importieren/exportieren

## Fuer AI ERP
Das Memory-Konzept uebersetzt sich zu:
- **Unternehmens-Wissensbasis**: Prozesswissen, Entscheidungshistorie, Lessons Learned
- **Kunden-Kontext**: Der Finance Agent "erinnert sich" an fruehere Buchungsentscheidungen
- **Audit-Trail**: Warum wurde eine Entscheidung getroffen? Memory liefert den Kontext
- **Aber**: Bilanzrelevante Daten gehoeren in eine **transaktionale Datenbank** (Postgres), nicht ins Memory
