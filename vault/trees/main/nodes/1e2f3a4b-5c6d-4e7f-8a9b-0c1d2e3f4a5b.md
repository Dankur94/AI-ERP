# Content-addressable / Idempotent

## Prinzip

Statt "fuehre Transaktion aus" eher "dieser Zustand soll existieren" — wiederholbar, ohne Seiteneffekte.

## Beispiele

- **Git:** Inhalte ueber Hashes adressiert, nicht ueber Pfade
- **IPFS:** Content-addressable Storage im Netz
- **Nix/Guix:** Reproduzierbare Builds ueber Hash-basierte Pakete
- **Terraform:** `terraform apply` zweimal → gleicher Zustand
- **REST PUT:** Idempotent per Definition

## SAP-Kontext

SAP-Transaktionen sind **nicht idempotent**: `VA01` (Auftrag anlegen) zweimal ausfuehren erzeugt zwei Auftraege. Das ist ein fundamentales Problem:
- Doppelbuchungen durch Bedienungsfehler
- Retry-Logik bei Netzwerkproblemen ist gefaehrlich
- Kompensationsbuchungen statt echtem Undo

## Vorteil

- **Fehlerresistenz:** Gleiche Aktion zweimal ausfuehren schadet nicht
- **Retry-sicher:** Bei Netzwerkproblemen einfach nochmal senden
- **Reproduzierbar:** Gleiche Eingabe → gleicher Zustand
- **Begrenzter Blast Radius:** Fehlbedienung erzeugt nicht unbeabsichtigt Duplikate

## Relevanz fuer AI ERP

- **Idempotenz als Default:** Alle Schreib-APIs idempotent (`PUT` mit Idempotency-Key)
- **Idempotency-Keys:** Client generiert UUID, Server dedupliziert
- **Content-adressierte Artefakte:** Dokumente, Berichte, Exporte ueber Hash identifiziert
- **Immutable Audit-Log:** Aenderungen als Append-Only-Events, nicht als Ueberschreiben
