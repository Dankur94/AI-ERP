# System-Architektur — 1 System statt 5 Inseln

## Was

Du designst wie die Tools des Kunden zusammenspielen. Nicht 5 isolierte Loesungen, sondern 1 kohaerentes System.

## Warum Claude das nicht kann

- Claude baut was man ihm sagt — Punkt fuer Punkt, Tool fuer Tool
- Claude denkt nicht: "Wie haengt dein Lager mit deinen Rechnungen mit deiner Buchhaltung zusammen?"
- Ohne Architekten baut der Kunde: WhatsApp-Bot + Excel-Parser + Invoice-Tool + Lager-App → 4 Inseln, 0 Integration

## Das Insel-Problem

Michelle baut mit Claude:
1. Montag: Lager-Tracker (Excel → App)
2. Mittwoch: Bestell-Tool (WhatsApp → Formular)
3. Freitag: Reporting-Dashboard

**Ergebnis:** 3 Tools die nicht miteinander reden. Bestellung geht rein → Lager weiss nichts davon. Gleicher Mist wie vorher, nur digitaler.

## Dein Mehrwert

Du siehst das Ganze:

```
Bestellung rein → Lager pruefen → Lieferung ausloesen → Rechnung erstellen → Buchhaltung
     |                |                  |                    |                  |
     +----------------+------------------+--------------------+------------------+
                              EIN System, EIN Datenfluss
```

## Methode

1. **Datenfluss-Map** — Welche Daten fliessen wohin?
2. **Single Source of Truth** — Wo lebt jede Information genau einmal?
3. **Integration Points** — Welche Systeme muessen miteinander reden?
4. **Build-Reihenfolge** — Was zuerst, damit spaeteres darauf aufbaut?

## Output

- System-Diagramm mit Datenfluss
- Technologie-Entscheidungen (Stack, Hosting, Datenbank)
- Build-Reihenfolge mit Abhaengigkeiten
- Kosten-Schaetzung fuer den Kunden
