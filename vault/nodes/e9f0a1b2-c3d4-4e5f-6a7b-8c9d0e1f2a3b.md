## 1. Was bauen wir

**Dokument rein → extrahieren → abgleichen → reporten → warnen — eine Pipeline, ein Tool, fuer kleine Firmen.**

Kein Wettbewerber hat die volle Pipeline fuer KMU:
- Tofu extrahiert, aber hoert danach auf
- SAP Concur matcht, aber nur fuer Konzerne
- Xero bucht, aber nur Buchhaltung
- Yonyou macht alles, aber komplex und teuer

**Wir:** Eine Pipeline, ein Login, ein Preis. Branchenuebergreifend. Sofort nutzbar.

6 Module in Reihenfolge:
```
1. Erfassen → 2. Finden → 3. Abgleichen → 4. Report → 5. Warnen → 6. Dashboard
```

## 1b. Gesetzte Entscheidungen

| Entscheidung | Gesetzt | Begruendung |
|---|---|---|
| **Interface: Web + App** | Browser (Haupt) + Mobile App (Ergaenzung) | Alle 4 Wettbewerber setzen auf Browser. Dashboard/Matching braucht Bildschirm. |
| **Backend-Sprache** | Python + FastAPI | AI-Kern braucht Python-Oekosystem. Schnellster Weg zum MVP. Go als Ergaenzung ab Stufe 3. |
| **Datenbank** | SQLite → Postgres ab Stufe 3 | 1 Datei, null Setup, reicht fuer 5-50 Kunden. Migration gut dokumentiert. Claude kennt SQLite perfekt. |
| **Frontend** | Vue.js | Leichtgewichtig, wenig Boilerplate, waechst mit Stufen. Kein Wegwerfcode. Yonyou nutzt es auch. |
| **On-Premise** | Spaeter, nur auf Nachfrage | Stufe 4. David Wong hat keinen Server. Aber als Option bewerben. |
| **WhatsApp** | Spaeter als Zusatzkanal | PDF per WhatsApp schicken → Ergebnis im Browser-Dashboard sehen. |
| **AI-Provider** | Claude API (Sonnet fuer Extraktion) | API in Stufe 1-2. Fine-tuning ab Stufe 3. Eigenes Modell ab Stufe 4. |
| **UI-Sprache** | Englisch | Kunden in Hong Kong. |
| **Trainingsdaten** | Korrekturen von Tag 1 speichern | Jede Korrektur = Trainingsmaterial. Unser Burggraben/Moat. ToS: "anonymized data to improve accuracy". |

## 8. Umsetzungs-Prozess (vor jedem Feature)

1. **Sagen** was umgesetzt werden soll und warum
2. **Optionen** eroertern in Frage-Antwort-Diskussion
3. **Entscheiden** gemeinsam mit dem Benutzer
4. **Festschreiben** in Master-MD als gesetzte Entscheidung
5. **Bauen** erst nach Freigabe

Nie direkt bauen. Immer erst diskutieren.

### YXZ-Check (bei jeder Entscheidung)
- **Y** — Wie hat der Wettbewerb (Yonyou/Tofu/SAP/Xero) es umgesetzt?
- **X** — Werden Inhalte der Master-MD angegriffen (Leitprinzip, Lessons, Stufenplan)?
- **Z** — Ist die Umsetzung vorteilhaft fuer Claude Coding? (nur bei Tech-/Library-Entscheidungen)

**Tech-Entscheidungen:** YXZ (alle drei)
**Modul-Umsetzung:** YX reicht (Z nur bei ungewoehnlicher Library-Wahl)

## 9. Code-Konventionen

- Python mit Type Hints (strict)
- Keine print() in Production
- Jede Funktion braucht Type Hints
- Keine Secrets im Code
- Tests fuer jede Business-Logik
- Commit-Messages: `feat:`, `fix:`, `refactor:`
