# Event-driven / Reaktiv

## Prinzip

Statt synchroner Transaktion ("T-Code aufrufen, warten") reagieren Systeme auf Ereignisse. CICS-Denken ist request/response; moderne Architekturen sind Strom-basiert.

## Beispiele

- **Event Sourcing:** Zustandsaenderungen als unveraenderliche Events gespeichert
- **CQRS:** Lesen und Schreiben getrennt optimiert
- **Kafka / Event Streams:** Asynchrone Nachrichtenverarbeitung
- **Reactive Extensions (Rx):** Observable-basierte Datenfluesse

## SAP-Kontext

SAP hat Event-Verarbeitung nachtraeglich eingefuehrt:
- **IDoc / ALE:** Asynchroner Datenaustausch zwischen SAP-Systemen
- **Business Events:** Trigger fuer Workflows
- **SAP Event Mesh:** Cloud-basiertes Event-Routing (neuere Ergaenzung)

Das zeigt: Das imperative Grundmodell reichte nicht aus. Events mussten nachgeruestet werden — was zu Inkonsistenzen und Komplexitaet fuehrt.

## IBM-Kontext

IBM hat mit **MQ Series** (heute IBM MQ) und **CICS Events** aehnliche Bruecken gebaut. Das Grundproblem ist dasselbe: Transaktionale Systeme sind synchron gedacht, asynchrone Realitaet erfordert Nachruesten.

## Vorteil fuer ERP

- **Entkopplung:** Module muessen nicht voneinander wissen
- **Skalierbarkeit:** Events koennen parallel verarbeitet werden
- **Audit:** Der Event-Log IST die Wahrheit (Event Sourcing)
- **Undo:** Events koennen rueckgaengig gemacht werden (Kompensations-Events)
- **Echtzeit:** Dashboards, Benachrichtigungen, Integrationen sofort

## Relevanz fuer AI ERP

- **Event-Bus als Kern-Infrastruktur:** Alle Zustandsaenderungen als Events
- **Event Sourcing fuer geschaeftskritische Domaenen:** Finanzen, Lager, Bestellungen
- **CQRS wo sinnvoll:** Lese-Modelle separat fuer Reporting/Dashboards
- **Webhook/Event-API:** Externe Systeme koennen auf Events reagieren
