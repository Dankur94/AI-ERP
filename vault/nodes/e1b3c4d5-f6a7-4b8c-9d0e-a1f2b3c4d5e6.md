# OpenClaw — Architektur-Lessons

**Quelle:** OpenClaw von Peter Steinberger (Open-Source AI Agent, 247k GitHub Stars)

## Lessons

| # | Lesson | Bedeutung fuer AI ERP |
|---|---|---|
| OC1 | **Lane Queue — 1 Queue pro Entity** | Jeder Tenant/Auftrag bekommt eigene Queue. Keine Race Conditions. |
| OC2 | **Human-Readable State** | Config, Audit-Trails in YAML/Markdown/JSONL. Keine Blackbox. |
| OC3 | **Channel Adapter Pattern** | Externe Quellen (APIs, Gateways) durch Adapter normalisieren. Ein internes Datenmodell. |
| OC4 | **Skills als Dateien, nicht Code** | Workflow-Anpassung ueber Text-Dateien statt Programmierung. Non-Devs koennen anpassen. |
| OC5 | **Modulare Prompts** | Jedes Modul bringt eigenen LLM-Kontext mit. Verhalten aendern ohne Code. |
| OC6 | **Memory-Schichten** | Session → Daily Logs → Long-term → Semantic Search. Wissen reift ueber Zeit. |
| OC7 | **Structured Keys fuer Multi-Tenancy** | `tenant:department:userId` — Isolation im Key-Design von Tag 1. |
| OC8 | **Hub-and-Spoke Gateway** | Ein zentraler Einstiegspunkt. Vereinfacht Auth, Logging, Debugging. |
| OC9 | **Defense in Depth** | Jeder Endpunkt braucht Auth, Plugins in Sandbox, Credentials in OS-Keystore. |
| OC10 | **Orchestration > Model** | Das Schwere ist nicht die AI, sondern alles drumherum. Solides Engineering ist das Fundament. |

**Referenz:** Snowboard-Shop-Analogie — OpenClaw loeste zuerst das eigene Problem (AI Agent fuer Entwickler), dann wurde es eine Plattform.
