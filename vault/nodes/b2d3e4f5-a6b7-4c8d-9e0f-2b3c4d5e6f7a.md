DDD wurde 2003 von Eric Evans beschrieben. Die Kernidee:

**Software soll die Realitaet des Geschaefts abbilden, nicht die Technik.**

Statt "users", "records", "tables" zu denken, denkt man in der Sprache des Kunden: "Auftrag", "Lieferant", "Rechnung", "Buchungssatz".

Drei Sauelen:
1. **Ubiquitous Language** — Entwickler und Fachexperten benutzen dieselben Begriffe. Im Code steht `Rechnung`, nicht `Invoice` oder `Document`. Kein Uebersetzen noetig.

2. **Strategisches Design** — WO die Grenzen zwischen Modulen liegen (Bounded Contexts). "Buchhaltung" und "Lagerverwaltung" sind getrennte Welten mit eigenen Regeln.

3. **Taktisches Design** — WIE innerhalb eines Moduls der Code strukturiert ist (Entities, Value Objects, Aggregates, Repositories, Domain Events).

Wichtig: DDD ist KEIN Framework. Es ist eine Denkweise. Man kann DDD in jeder Sprache und mit jedem Framework umsetzen.
