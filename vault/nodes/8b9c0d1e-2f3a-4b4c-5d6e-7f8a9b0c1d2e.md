# Deklarativ vs. Imperativ

## Prinzip

Nicht "fuehre Schritt A, B, C aus", sondern "der Zielzustand ist X" — das System findet den Weg.

## Beispiele

| Deklarativ | Imperativ |
|---|---|
| SQL: `SELECT * WHERE ...` | Cursor-basiertes Iterieren |
| Terraform: `resource "server" { count = 3 }` | Manuell 3 Server provisionieren |
| React: `<Component state={x} />` | DOM manuell manipulieren |
| Kubernetes: `replicas: 3` | Container einzeln starten |

## SAP-Kontext

SAP ist durchgehend imperativ: `VA01` → Felder ausfuellen → Sichern. Man beschreibt nicht den Zielzustand ("es soll ein Auftrag mit diesen Daten existieren"), sondern den Weg dorthin.

Das macht SAP fragil bei Aenderungen: Wenn sich der Prozess aendert, muessen alle T-Code-Abfolgen angepasst werden. Ein deklaratives System waere stabiler: "Zielzustand X" bleibt gleich, auch wenn sich der Weg aendert.

## Vorteil

- **Stabilitat:** Was vom Wie getrennt
- **Idempotenz:** Zweimal ausfuehren hat kein anderes Ergebnis
- **Compose-faehig:** Deklarative Beschreibungen lassen sich zusammenfuehren

## Relevanz fuer AI ERP

- **Konfiguration deklarativ:** YAML/JSON-basierte Modul- und Systemkonfiguration
- **Geschaeftsregeln deklarativ:** "Bestellungen ueber 10.000 EUR brauchen Freigabe" statt prozeduraler Prueflogik
- **API deklarativ:** `PUT /order/123 { status: "approved" }` statt `POST /order/123/approve`
- **Plan-vs-Apply:** Wie Terraform — System zeigt an, was sich aendern wuerde, Nutzer bestaetigt
