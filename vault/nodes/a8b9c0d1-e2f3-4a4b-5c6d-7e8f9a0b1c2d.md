# Lessons — Ranking nach Leitprinzip

**Leitprinzip:** "Heute fuer 5-15 Leute bauen, aber so, dass es bis Enterprise skaliert."

```
Stufe 1 (jetzt)       Stufe 2 (12M)        Stufe 3 (24M)        Stufe 4 (48M)
Custom pro Kunde  →   Wiederverwendbare  →  SaaS-Produkt     →   Enterprise
                      Module
```

---

## Treffen den Kern (Lesson IST das Leitprinzip)

| Rank | Lesson | Warum |
|---|---|---|
| **1** | **L4** — Komplexitaet progressiv freischalten | Das IST der Stufenplan in einer Lesson. Einfach starten, graduell wachsen. |
| **2** | **L1** — Kein Enterprise-Fork | Garantiert dass Stufe 1-4 derselbe Code bleibt. |
| **3** | **L3** — Kundenisolation by Design | Muss in Stufe 1 rein, sonst ist Stufe 2-4 ein Totalumbau. |
| **4** | **L2** — Geteiltes Datenmodell | Haelt alle Stufen zusammen. Ein Datenmodell von Anfang bis Ende. |

## Unterstuetzen das Leitprinzip

| Rank | Lesson | Warum |
|---|---|---|
| **5** | **L7** — Produkt folgt den Daten | Stufe 1 = Kunden beobachten → Stufe 2 = Module daraus bauen. Genau der Plan. |
| **6** | **L12** — Human-Readable State | Einfach in Stufe 1, bleibt inspizierbar bis Stufe 4. Kein Lock-In. |
| **7** | **L8** — Eigenes Problem zuerst loesen | Stufe 1 = ein echtes Problem loesen, nicht eine Plattform traeumen. |
| **8** | **L5** — Adapter Pattern | Ermoeglicht Stufe 3-4 (verschiedene Systeme andocken), schadet nicht in Stufe 1. |
| **9** | **L16** — Orchestration > Model | Haelt den Fokus: erst solide bauen, dann AI drauf. Passt zum Stufenplan. |
| **10** | **L10** — Revenue trackt Kundenerfolg | Geschaeftsmodell-Entscheidung die in jeder Stufe funktioniert. |
| **11** | **L14** — Modulare Prompts | Hilft ab Stufe 2 wenn Module wiederverwendbar werden. |
| **12** | **L13** — Skills als Dateien | Hilft ab Stufe 2-3 fuer Customization ohne Entwickler. |

## GREIFEN DAS LEITPRINZIP AN

Diese Lessons verleiten dazu, **heute fuer Enterprise zu bauen statt fuer 5-15 Leute:**

| Rank | Lesson | Konflikt |
|---|---|---|
| **13** | **L6** — API-Versionierung als Burggraben | API-Versionierung fuer einen Kunden mit 8 Leuten? **Overengineering in Stufe 1.** Erst ab Stufe 3-4. |
| **14** | **L11** — Queue pro Entity | Queue-System fuer 5 Kunden mit je 10 Rechnungen/Tag? **Unnoetige Komplexitaet.** Erst ab Stufe 3. |
| **15** | **L9** — Sandboxing fuer Drittanbieter | In Stufe 1 gibt es keine Drittanbieter. **Loest ein Problem das noch nicht existiert.** Stufe 4. |
| **16** | **L15** — Memory-Schichten (4 Layer) | 4-Layer-Memory mit Dreaming fuer einen Steuerberater-Export? **Massiv overbuilt.** Fruehestens Stufe 3. |

## Faustregel

```
L4, L1, L3, L2       ← DAS SIND dein Leitprinzip (Pflicht ab Tag 1)
L7-L14                ← Unterstuetzen es (einbauen wenn passend)
L6, L11, L9, L15      ← GEFAEHRLICH: Verleiten zum Overbuilding
```

Wenn eine Lesson dich dazu bringt, etwas fuer Stufe 3-4 zu bauen waehrend du in Stufe 1 bist — **ignoriere sie bis du dort bist.**
