# Session 2026-04-25

## Was gemacht

### L3: Kundenisolation (tenant_id)
- tenant_id Spalte in invoices-Tabelle eingefuehrt
- Alle DB-Queries (get, list, update, search, match, report) filtern nach tenant_id
- Automatische Migration fuer bestehende DBs (_migrate_tenant_id)
- Default-Tenant "default" fuer Stufe 1, spaeter aus Auth-Token
- PDF-Download prueft ebenfalls tenant_id

### Modul 1: Claude Vision API
- extraction.py komplett umgebaut mit Adapter Pattern (L5)
- Echter Claude Vision API Call wenn ANTHROPIC_API_KEY gesetzt
- Fallback auf Mock-Daten wenn kein Key (Entwicklung)
- JSON-Parsing mit Code-Fence-Handling
- Config-Loader (config.py): YAML-Datei oder Umgebungsvariablen
- config.example.yaml erweitert mit Anthropic-Sektion
- Neue Dependencies: anthropic, pyyaml

### Modul 2: Suche & Filter
- Backend: GET /api/invoices/search mit Parametern q, supplier, status, date_from, date_to, currency
- Alle Filter kombinierbar, tenant-isoliert
- Frontend: SearchView.vue mit Suchleiste (Live-Suche, Debounce 300ms)
- Filter-Dropdowns fuer Status, Lieferant, Datum, Waehrung
- Ergebnistabelle mit Klick-Navigation
- Route /search, Navigation im Header

### Modul 3: Matching / Abgleich
- Backend: POST /api/matches, GET /api/matches, GET /api/matches/{id}, DELETE /api/matches/{id}
- Matching-Service (services/matching.py): Feldvergleich + Positionen-Vergleich
- Abweichungserkennung: match, mismatch, missing_a, missing_b
- Positionen-Matching: exakt + partiell (Beschreibung)
- Frontend: MatchView mit Dropdown-Auswahl, Vergleichstabelle, farbiger Markierung
- Match-Verlauf mit Loeschfunktion
- Route /matches

### Modul 4: Reports
- Backend: GET /api/reports/summary (Statistiken), GET /api/reports/export/csv
- Summary: Anzahl, Summen, Aufschluesselung nach Status/Lieferant/Waehrung
- CSV-Export mit Datumsfilter
- Frontend: ReportView mit 4 KPI-Karten, Tabellen, CSV-Download-Button
- Route /reports
- L4: CSV fuer Stufe 1, PDF/Excel spaeter

### Tests
- 58 Tests, alle bestanden
- test_database.py: CRUD, Tenant-Isolation, Suche
- test_api.py: Alle Endpoints
- test_extraction.py: Mock + Claude-Response-Parsing

### Lessons befolgt
- L1: Kein Fork — alles in einer Codebasis, ein Backend, ein Frontend
- L2: Geteiltes Datenmodell — selbe DB, alle Module teilen invoices/line_items
- L3: Kundenisolation — tenant_id in allen Tabellen und Queries
- L4: Progressiv — LIKE statt FTS5, CSV statt PDF-Export, Default-Tenant statt Auth
- L5: Adapter Pattern — Extraction-Service austauschbar, Matching-Service modular
- L12: Human-Readable — Config in YAML
- L16: Orchestration > Model — solides CRUD, AI nur in Extraktion
- NICHT gebaut: L6 (keine API-Versionierung), L9 (kein Sandboxing), L11 (keine Queues), L15 (kein 4-Layer Memory)

## Starten

```bash
# Dependencies installieren (neue Pakete):
source .venv/Scripts/activate
pip install -r src/api/requirements.txt

# Backend (Terminal 1):
bash start-backend.sh
# → http://localhost:8000

# Frontend (Terminal 2):
bash start-frontend.sh
# → http://localhost:5173

# Tests:
source .venv/Scripts/activate
python -m pytest tests/ -v

# WICHTIG: venv nutzt Windows Python 3.12
```
